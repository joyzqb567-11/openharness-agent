# agent.py 中文讲解

本文解释文件：`H:\codexworkplace\sofeware\OpenHarness-main\learning_agent\core\agent.py`。

## 一句话理解

`agent.py` 是这个教学 agent 的“总调度员”。用户输入一句话后，它负责准备上下文、调用模型、接收模型想用的工具、执行工具、把工具结果喂回模型，最后把答案返回给用户。

你可以把它想象成一个前台经理：

- 模型是“大脑”，负责判断下一步该说话还是该用工具。
- 工具是“手脚”，负责读文件、写文件、跑命令、查 MCP、控制桌面等。
- 权限确认是“门禁”，写文件、执行命令、控制桌面前要先问用户。
- 事件流和日志是“监控摄像头”，记录每一步发生了什么，方便恢复和排查。

## 文件整体结构

这个文件有四层：

1. 第 1 到 220 行左右：导入区。
   这里把核心模块、模型模块、MCP 模块、工具模块、浏览器模块、任务模块、观测模块和 CLI 模块接进来。它还有很多 `try / except ModuleNotFoundError`，目的是兼容两种运行方式：作为包运行，以及直接从 bat 脚本运行。

## 最重要的主流程：run_events()

`run_events()` 是这个 agent 真正跑起来的地方。它不是只返回一个字符串，而是一边运行一边产生事件，所以终端、HTTP bridge、日志和恢复系统都能观察到过程。

它的大致流程是：

1. 创建 `session_id` 和 `run_id`。
   这样一次对话能被日志、事件和恢复文件串起来。

2. 建立 transcript 和 session store。
   transcript 记录事件流水，session store 保存可恢复摘要。

3. 根据用户输入识别特殊场景。
   例如真实 Chrome、可见浏览器查询、桌面任务、计划确认等。

4. 调用 `_build_initial_messages()` 生成模型上下文。
   这里会把系统提示词、项目记忆、用户输入、工具提示等装配给模型。

5. 进入多轮循环。
   每一轮先检查是否需要压缩上下文，再把 messages 和当前可见工具 schema 交给模型。

6. 接收模型响应。
   如果模型只是回答文本，没有工具调用，就保存 session 并结束。

7. 如果模型请求工具调用，就执行工具。
   工具执行结果会被追加回 messages，下一轮模型就能根据工具结果继续判断。

8. 达到最大轮次或出现异常时，输出安全停止或失败事件。

通俗地说：`run_events()` 就是“用户一句话 -> 模型思考 -> 工具干活 -> 模型继续思考 -> 最终回答”的发动机。

## LearningAgent 初始化在准备什么

`LearningAgent.__init__()` 会把 agent 运行需要的东西全部装好：

- `self.model`：模型客户端。
- `self.workspace`：允许操作的工作区。
- `self.ask_permission`：权限确认函数。
- `self.mcp_tool_registry`：MCP 工具注册表。
- `self.loaded_tool_names`：当前已经解锁的延迟工具。
- `self.tool_policy` 和 `self.tool_policy_context`：工具可见性和风险策略。
- `self.computer_use_controller`：桌面控制后端。
- `self.observation_events`：结构化观察记录。
- `self.task_runs`、`self.team_peers`、`self.cron_records`、`self.monitor_records`：任务、协作、定时和监控状态。
- `self.memory_path`、`self.todo_path`、`self.skills_path`：长期记忆、待办状态和 skill 目录。

它做的不是“回答问题”，而是把回答问题需要的运行环境先搭好。

## 工具池和 Tool Loop

模型不能直接碰电脑，它只能发出类似“我要调用 read_file，参数是 path=xxx”的工具请求。`agent.py` 的工作是：

1. 决定模型当前能看到哪些工具。
2. 把工具 schema 发给模型。
3. 接收模型返回的 `ToolCall`。
4. 调用 `_execute_tool()` 或工具编排器执行。
5. 把工具结果作为 `tool` 消息放回上下文。

这里有一个关键设计：工具不是一次性全部暴露。很多工具是 deferred，也就是“需要搜索或读取 skill 后才加载”。这样可以减少模型注意力漂移，也能节省 token。

## MCP 在这里做什么

MCP 可以理解为“外部工具接入协议”。`agent.py` 通过 `McpToolRegistry` 管理外部 server：

- 启动 MCP server 前会先走权限确认。
- 启动成功后，把外部工具转成模型能理解的 schema。
- 调用外部工具时，先判断风险，再执行。
- MCP 资源、prompt、stream 也有单独的读取函数。

也就是说，MCP 不是模型自己连外部工具，而是 agent 作为中间人，负责发现、审批、调用和记录。

## 权限系统为什么很重要

这个文件里有很多 `ask_permission`。它的意义是：模型可以建议操作，但不能默默改你电脑。

典型需要确认的动作包括：

- 写文件。
- 执行 shell 命令。
- 写长期记忆。
- 调用高风险 MCP 工具。
- 控制桌面窗口。
- 启动或停止后台进程。

第 3629 行以后的权限函数还会把权限请求转成结构化事件，方便外部验收系统判断“到底是哪一个工具、哪些参数请求了授权”。

## 文件工具和原子工具

`_read_file()`、`_write_file()`、`_append_memory()` 是早期教学版工具。

后面又出现 `_read_atom()`、`_write_atom()`、`_edit_atom()`、`_bash_atom()`，这些是更简洁的“四原子工具”：

- read：读。
- write：写。
- edit：改。
- bash：执行命令。

它们的目标是让工具面更小、更清楚，减少模型在几十个工具名里迷路。

## 任务、团队、计划模式

`agent.py` 还支持更长任务：

- `task`：启动子任务或子 agent。
- `task_list`、`task_get`、`task_update`：管理任务。
- `team_create`、`send_message`：教学版多 agent 协作。
- `enter_plan_mode`、`exit_plan_mode`、`verify_plan_execution`：计划模式。

这些功能的目的，是让 agent 不只会“一问一答”，还能把较大的目标拆开、记录、恢复和验证。

## 桌面控制相关代码

`_computer_status()`、`_computer_observe()`、`_computer_action()`、`_computer_use_mode()` 和 `_computer_use_compat()` 是桌面控制入口。

它们不是直接乱点屏幕，而是通过 `ComputerUseController` 和后续的通用桌面运行时做几件事：

- 先观察当前桌面或窗口。
- 判断目标和动作风险。
- 高风险动作请求权限。
- 执行动作后记录结果和截图证据。

这也是为什么项目里有大量 Phase 相关注释，因为桌面控制功能是逐步验收出来的。

## 为什么这个文件看起来特别长

它长的原因不是单纯代码乱，而是它现在承担了很多“兼容入口”和“总调度入口”职责：

- 要兼容包运行和 bat 脚本运行。
- 要保留旧测试入口。
- 要承接新拆分出去的模块。
- 要把模型、工具、MCP、桌面控制、任务、日志串起来。

从架构上看，它正在变成“薄总线”：具体实现逐步拆到 `core/`、`models/`、`mcp/`、`tools/`、`computer_use/`、`app/` 等模块里，`agent.py` 保留主要编排逻辑。

## 初学者记忆法

读这个文件时，不要从第 1 行硬啃到最后。按下面顺序看：

1. 先看 `LearningAgent.__init__()`：它准备了哪些能力。
2. 再看 `run_events()`：一次请求怎么跑完。
3. 再看 `_build_initial_messages()`：模型一开始看到什么。
4. 再看 `_available_tool_schemas()` 和 `_execute_tool()`：工具如何出现和执行。
5. 最后按兴趣看 MCP、任务、桌面控制、文件工具。

最核心的理解只有一句：`agent.py` 不是模型本身，它是让模型能够安全、可观察、可恢复地使用工具的运行时外壳。
