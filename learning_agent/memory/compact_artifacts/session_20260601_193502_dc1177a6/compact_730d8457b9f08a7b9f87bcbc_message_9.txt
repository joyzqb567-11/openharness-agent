# Real Chrome

入口顺序：先 read `learning_agent/skills/tool_list.md`，再 read 本文件。

本 skill 只判断是否进入真实 Chrome 高风险能力。需要细节时，再读取：

- `learning_agent/skills/real_chrome/rules/profile_safety.md`
- `learning_agent/skills/real_chrome/rules/search_task_workflow.md`

注意：`browser_launch_visible` 只会启动独立 Chromium 的肉眼可见窗口，用于验收页面操作能力；它不是用户日常 Chrome profile，也不会自动携带用户登录态。只有用户明确要求真实 Chrome、当前 Chrome 或登录态时，才走 `browser_profile_status` -> `browser_connect_real_chrome` 的真实 Chrome workflow。

读取本文件成功后，运行时会准备 `real_chrome` 和后续页面操作所需的 `browser_automation` MCP 工具；真实连接仍必须先执行 `browser_profile_status`，状态检查完成后才可调用 `browser_connect_real_chrome`。

双轨规则：不要直接选择 provider，不要发明 `chrome_extension_open`、`real_chrome_cdp_click`、`visible_chromium_type` 这类工具；模型只调用统一 `browser_*` 工具，底层由 `BrowserProviderRouter` 根据当前 Chrome、登录态、CDP 调试和 provider 健康状态选择路线，并把选择写入 event log。

边界：
- 不要用搜索、fetch、普通独立 Chromium 替代用户明确要求的真实 Chrome。
- 真实 profile 可能包含 cookies、localStorage、sessionStorage、token 和账号状态。
- 连接真实 Chrome 后，继续使用解锁后的浏览器打开、点击、输入、截图和页面检查工具操作同一个真实浏览器会话。
- 连接真实 Chrome 后，任何 `browser_click`、`browser_type`、`browser_type_secret`、`browser_press_key`、`browser_upload_file` 前，必须先调用 `browser_tabs_context` 确认 active tab、URL、标题和 page_id；切换、新建、关闭或导航标签页后也要重新调用。
- 若任务要求当前 Chrome 插件路线，先用 `browser_extension_status` 查看 Chrome extension/native host 是否连接；Stage 5 插件 provider 只支持只读 `browser_tabs_context` 和 `browser_snapshot`，不能执行点击输入。
- 真实 Chrome 下默认不要读取 cookies、localStorage、sessionStorage、token、password，也不要用 `browser_evaluate` 绕过页面安全边界。
- 需要更接近 ClaudeCode 站点级权限时，先用 `browser_site_grant` 授权 origin，再用 `browser_site_grant` 的 enable_strict 开启严格站点边界。
- 真实 Chrome 登录态动作默认只审计不自动回放；`browser_replay` 在真实 Chrome 模式下只能用于 dry_run 查看历史动作。
- 页面失败时先用 `browser_recover_page` 恢复当前真实浏览器页面；selector 不稳定时用 `browser_visual_locate` 找中心点，再用坐标点击。
